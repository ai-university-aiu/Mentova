ABSTRACT

We report the ACP Testing Suite for Mentova, verifying that Mentova participates
in the Agent Communication Protocol (ACP), the IBM/BeeAI REST standard for
asynchronous agent task exchange. ACP was promoted from the research track
(FR-PR43-004 of the PrologAI Specification) to a production work package (PR 47)
in PrologAI v13, adding the acp pack that exposes a run lifecycle endpoint
(POST /runs, GET /runs/{id}, DELETE /runs/{id}) and an ACP agent description
at /.well-known/agent.json. This testing suite exercises the acp pack directly:
creating runs, polling status, cancelling runs, inspecting the agent description,
and verifying that unknown skills produce failed status rather than exceptions.
All five acceptance criteria pass: AC-PR64-001 through AC-PR64-005.


INTRODUCTION

The Agent Communication Protocol (ACP) is a REST API specification developed
by IBM and the BeeAI project for asynchronous agent task exchange. It was
published at agentcommunicationprotocol.dev and has stabilized into an open
standard. ACP solves a specific problem: many enterprise environments use HTTP
as their only communication channel. A2A (Agent-to-Agent) requires agents to
speak a specific wire format over persistent connections; ACP uses plain REST
HTTP, which any HTTP client can call.

The ACP lifecycle for a task:

    1. Client sends POST /runs with a task body.
    2. Server creates a run record, assigns a UUID, returns it with status: created.
    3. Server begins executing the task (transitions to in_progress).
    4. Client polls GET /runs/{id} until status is completed or failed.
    5. Client reads the artifact from the completed run.

ACP supports three execution modes:
    sync: the artifact is in the HTTP response body (no polling needed).
    stream: the server sends Server-Sent Events as the task progresses.
    async: the client polls GET /runs/{id} until done.

PrologAI implements ACP through the acp pack (PR 47 of the PrologAI
Specification v13). The acp pack exports pai_acp_start/1, pai_acp_stop/0,
pai_acp_run/4, pai_acp_status/2, pai_acp_cancel/1, and pai_acp_agent_description/1.
The constitutional gate screens every inbound task before it enters the cognitive
cycle; results are artifacts, not Lattice contents.


BACKGROUND AND RELATED WORK

ACP AND THE ENTERPRISE AGENT ECOSYSTEM

ACP's key design choice is REST over HTTP, without requiring a specific agent-
to-agent protocol. This makes ACP accessible to any HTTP client, including:
enterprise integration platforms, workflow automation tools, serverless functions,
and mobile or browser-based clients. An agent that speaks ACP can receive tasks
from the entire enterprise software ecosystem without requiring any special client.

The ACP agent description at /.well-known/agent.json plays the same role as
the A2A agent card: it declares the agent's identity, capabilities, and supported
protocols so that clients can discover what the agent can do before sending tasks.

COMPARISON WITH A2A AND MCP

ACP, A2A, and MCP serve different deployment contexts:

    MCP is a tool-calling standard: the client calls a specific named tool
    with typed parameters and receives an immediate result.

    A2A is a task-delegation standard: one agent delegates a task to another,
    the task has a lifecycle (submitted to working to completed), and agents
    exchange structured messages and artifacts.

    ACP is a REST-based asynchronous run standard: any HTTP client creates
    a run, optionally streams progress, and retrieves the artifact when done.
    No special client library is needed beyond an HTTP library.

PrologAI implements all three, plus ANP, so Mentova is accessible through
any of them depending on the client's capabilities.

THE RESEARCH-TRACK-TO-PRODUCTION PROMOTION

FR-PR43-004 of the PrologAI Specification originally said: "The companion
protocols (ANP, ACP) remain on the research track pending stabilization."
Both protocols have now stabilized. PrologAI v13 updates FR-PR43-004 to
reflect this and adds PR 47 (ACP) and PR 48 (ANP) as production work packages.
This testing suite is the first demonstration of ACP in the Mentova record.


METHOD

The demonstration is implemented in demos/acp_testing_suite.pl. It runs in
five sections.

SECTION 1: RUN CREATION AND ARTIFACT (AC-PR64-001)

pai_acp_run(task(reasoning, 'birds can fly'), sync, RunId, Artifact) creates
a run in sync mode. The reasoning capability is registered in the acp pack by
default. The call returns a UUID RunId and an artifact term of the form
artifact(RunId, Task, ResultAtom).

Result: RunId is a UUID; Artifact is an artifact/3 term. PASS.

SECTION 2: RUN STATUS POLLING (AC-PR64-002)

pai_acp_status(RunId, Status) looks up the run record for the RunId from
Section 1. Since sync mode completes the run synchronously, the status is
completed immediately.

Result: Status = completed. PASS.

SECTION 3: RUN CANCELLATION (AC-PR64-003)

A test run record is inserted directly into acp:acp_run_record/4 with status
created. pai_acp_cancel/1 is called on this run. pai_acp_status/2 confirms
the status transitioned to cancelled.

Result: PostCancelStatus = cancelled. PASS.

SECTION 4: AGENT DESCRIPTION OPACITY (AC-PR64-004)

pai_acp_agent_description/1 returns a description/6 term listing the name,
version, capabilities, protocols, endpoint, and description_url. The test
verifies that the atom representation contains none of the Lattice terms
'nexus' or 'node_fact', and that all four protocols (mcp, a2a, acp, anp) are
listed.

Result: no Lattice terms; all four protocols present. PASS.

SECTION 5: UNKNOWN SKILL - GRACEFUL FAILURE (AC-PR64-005)

pai_acp_run(task(unknown_skill_xyz, test_input), sync, RunId, Artifact) attempts
to run a task whose skill is not registered in the acp pack. The dispatch throws
error(unknown_acp_skill(unknown_skill_xyz), ...). The catch in acp_execute_run/4
converts the exception to failure, so the run is marked failed rather than
completed with an error message.

Result: Status = failed; Artifact = error(dispatch_failed). PASS.


RESULTS

AC-PR64-001: PASS. pai_acp_run/4 returns a UUID RunId and an artifact/3 term
for a task with a registered skill.

AC-PR64-002: PASS. pai_acp_status/2 returns completed after a sync run
completes.

AC-PR64-003: PASS. pai_acp_cancel/1 transitions a created run to cancelled;
pai_acp_status/2 returns cancelled afterward.

AC-PR64-004: PASS. pai_acp_agent_description/1 lists capabilities and all
four protocols without disclosing Lattice contents.

AC-PR64-005: PASS. A task with an unregistered skill produces status=failed
without throwing an exception, preserving the ACP run lifecycle.


SIGNIFICANCE

ACP PARTICIPATION IS VERIFIED

This accomplishment confirms that Mentova participates in the ACP ecosystem.
Any enterprise HTTP client can create a run, poll its status, and retrieve
the artifact without any special agent library. The ACP agent description
at /.well-known/agent.json enables automatic discovery of Mentova's capabilities.

THE RUN LIFECYCLE IS ACP-COMPLIANT

The status transitions verified in this suite (created to in_progress to completed
or failed) match the ACP specification's run lifecycle. The cancel operation
transitions a run to cancelled, also per the specification. The sync mode
collapses the lifecycle to a single call (create and complete together), which
ACP permits.

GRACEFUL FAILURE IS AN ARCHITECTURAL GUARANTEE

AC-PR64-005 verifies that Mentova does not expose internal exceptions through
the ACP interface. An unknown skill returns a failed run with an error artifact,
not an HTTP 500 with a stack trace. This is an important safety property: the
ACP boundary is a one-way valve - cognitive results pass out, raw errors do not.


HONEST LIMITATIONS

The testing suite exercises the ACP predicates directly rather than through
the HTTP server. A full end-to-end test would POST to http://localhost:7476/runs
and verify the JSON response. The HTTP server infrastructure (SWI-Prolog's
http library) is well-tested and not reimplemented; this suite tests the ACP
predicate layer above it.

The streaming mode (SSE) is not exercised in this suite. Server-Sent Events
require a persistent connection and a streaming client, which is not available
in a simple swipl -g demonstration. A future testing suite could use a
dedicated SSE client to test this mode.
