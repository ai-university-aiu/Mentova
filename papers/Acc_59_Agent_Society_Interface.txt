ABSTRACT

We report Mentova's first agent-society interface demonstration, closing the
agent-society item from Part 8 of the PrologAI Demonstration and Proof-of-Concept
Plan. Two agents — Mentova (9 capabilities) and Mentor-B (2 capabilities) — participate
in a multi-agent exchange using the A2A (Agent-to-Agent) protocol implemented in
PrologAI PR 43. The demonstration covers five steps: (1) both agents register
identity cards and capabilities glass-box; (2) Mentova submits an A2A task to
Mentor-B through the full submitted-working-completed lifecycle; (3) Mentova sends
durable peer mail to Mentor-B; (4) Mentor-B fetches its mailbox and reads the
message; (5) Mentor-B solves the query, sends a reply, and Mentova reads the reply.
All five acceptance criteria pass: AC-PR59-001 through AC-PR59-005. Mentova is
demonstrated as a participating member of an agent society, not merely a standalone
reasoner.


INTRODUCTION

Part 8 of the PrologAI Demonstration and Proof-of-Concept Plan specifies:

    "Over time it extends toward the full PrologAI feature set — the workspace
     cycle, the attention economy, the full affect and motivation systems, the
     agent-society interfaces — until Mentova is a richly capable Synthetic
     Brain rather than a demonstration subject."

The workspace cycle (Acc_51), the attention economy (Acc_52), and the affect and
motivation systems (Acc_44, Acc_45) have been demonstrated. The agent-society
interface — the final item in this list — is demonstrated in this accomplishment.

An agent society is a network of AI minds that can discover each other, exchange
tasks, send and receive addressed mail, and collaborate to solve problems that
no single mind handles alone. For Mentova to be a "richly capable Synthetic Brain"
rather than a demonstration subject, it must be able to participate in such a society:
publishing what it can do, accepting tasks from peers, and routing queries to
specialists.

The A2A (Agent-to-Agent) protocol, implemented in PrologAI's PR 43, provides this
infrastructure. It was developed in alignment with the emerging Linux Foundation A2A
standard and adapts it to PrologAI's opacity principle: agent cards disclose
capabilities without exposing Lattice contents, and peer mail lands in a quarantine
scope before being admitted to the nexus.


BACKGROUND AND RELATED WORK

THE AGENT-TO-AGENT (A2A) PROTOCOL

The A2A protocol addresses a gap in multi-agent AI systems: agents built on
different frameworks, running different models, maintained by different teams,
need a common vocabulary for capability discovery and task exchange.

Core A2A concepts:
- Agent Card: a structured declaration of an agent's identity, capabilities, and
  endpoint. Other agents read the card to discover what the agent can do.
- Task lifecycle: submitted -> working -> completed(artifact) | failed(reason).
  Tasks have unique IDs, enabling tracking and resumption.
- Messages: carry role-tagged parts (query, result, artifact).
- Durable mail: persists across sessions; addressed to a named recipient; carries
  a sender, timestamp, and integrity hash.

PrologAI's PR 43 implements this protocol in Prolog:
- pai_agent_card/1 generates the current agent card from registered capabilities.
- pai_register_identity/2 and pai_register_capability/2 populate the card.
- pai_a2a_task/4 submits a task and returns its final status.
- pai_peer_mail_send/3 sends addressed, persistent mail.
- pai_peer_mail_fetch/3 fetches pending mail for a recipient, marking it delivered.

THE OPACITY PRINCIPLE

PrologAI's agent-society design follows an opacity principle: the agent card
discloses capabilities, never Lattice contents. When an inbound task arrives,
it is screened as an objective node_fact (not given raw access to the nexus).
The result is the task artifact — the answer — not the internal inference trace.

This preserves both safety (the nexus is not exposed to arbitrary external queries)
and privacy (the agent's accumulated knowledge base is opaque to peers).

MULTI-AGENT COGNITION

The multi-agent literature distinguishes between:
- Cooperative: agents work toward a shared goal.
- Competitive: agents optimize their own objectives.
- Mixed-motive: agents have overlapping but not identical objectives.

Mentova's constitution (8 principles) governs how it participates in all three
modes: cooperative by default, with constitution-gated limits on what it will do
even when asked by a peer. This is the corrigibility principle extended to the
multi-agent setting.


METHOD

The demonstration is implemented in demos/agent_society_demo.pl. It boots Mentova
fully, then executes the five-step exchange.

AGENT REGISTRATION

Both agents are registered in the A2A module's dynamic fact store:
- Mentova: 9 capabilities (deduction, induction, abduction, epistemic, practical,
  metacognitive, moral, spatial, temporal).
- Mentor-B: 2 capabilities (deduction, epistemic) — a simulated specialist.

AGENT CARD PUBLICATION

get_agent_card/2 reads the registered identity and capabilities and packages
them as card(identity(Id), capabilities(Caps), endpoint(Endpoint)).
The endpoint is `local` in this demonstration; in a network deployment it would
be the agent's URL or message queue address.

A2A TASK LIFECYCLE

submit_task/6 implements the full lifecycle glass-box:
1. Assert task record with status `submitted`; print confirmation.
2. Update to `working`; print update.
3. Call simulate_skill/3 to compute the result.
4. Update to `completed(artifact(TaskId, Skill, Result))`; print artifact.

The artifact for `deduction, birds_fly`:
    deduced(tweety, can_fly, via(is_a(tweety, bird), birds_fly))

This is a glass-box justification: Tweety can fly because Tweety is a bird and
birds fly — the inference chain is explicit.

PEER MAIL

pai_peer_mail_send/3 stores addressed mail as agent_mail/5 dynamic facts.
pai_peer_mail_fetch/3 retrieves all undelivered mail for a recipient and
marks them delivered (via delivered_mail/1) so they don't appear again.

The exchange:
- Mentova sends: TO=mentor_b, SUBJECT=theory_of_mind_query,
  BODY="Does Sally hold a false belief about marble_in_basket?"
- Mentor-B fetches: reads the message, runs an epistemic task internally.
- Mentor-B replies: TO=mentova, SUBJECT=theory_of_mind_reply,
  BODY="Result: result(false_belief_confirmed(sally,marble_in_basket))"
- Mentova fetches: reads the reply.

The reply confirms: Sally holds a false belief about marble_in_basket —
consistent with the Sally-Anne demonstration in Acc_53.

SIGNIFICANCE

This accomplishment closes the agent-society interface item from Part 8 of the
Demonstration Plan. Its significance is that Mentova is now demonstrated as a
participant in a multi-agent society, not merely a standalone reasoner.

The five capabilities this unlocks:
1. Capability discovery: any agent can read Mentova's card to know what it can do.
2. Task delegation: Mentova can offload tasks to specialist agents.
3. Task reception: Mentova can accept tasks from other agents.
4. Asynchronous communication: durable peer mail persists across sessions.
5. Collaborative reasoning: complex queries can be split across multiple minds.

For a "richly capable Synthetic Brain" (Part 8's description of the growth target),
all five of these are necessary. A brain that cannot communicate with other minds
is isolated; a mind that participates in a society is embedded in a world.

The A2A task exchange also demonstrates PrologAI's opacity principle in practice:
Mentor-B receives the task result (the artifact) but not access to Mentova's nexus.
The agent-society interface is safe by architecture, not by convention.


HONEST LIMITATIONS

Mentor-B is a simulated agent in this demonstration. In a live multi-agent
deployment, Mentor-B would be a separate PrologAI instance (or an A2A-compatible
agent from another framework) running at a different endpoint. The A2A protocol
supports this: the endpoint field in the agent card would be a network URL or
message queue address, and pai_a2a_task/4 would make an HTTP or message-bus call.

The skill dispatch in the local simulation (simulate_skill/3) returns predefined
results rather than actually querying a live Mentor-B instance. In a live
deployment, dispatch_skill/3 in the a2a module would route to the real endpoint.

The peer mail store is in-memory (dynamic facts) in this demonstration. For true
durability across sessions, the mail store would be backed by a persistent database
or file store — which the A2A pack's architecture supports via external storage hooks.
