ABSTRACT

We report Rung 17 of Mentova's 48-rung reasoning ladder: non-monotonic
(defeasible) reasoning. Mentova applies the default rule "birds fly" to a
canary and returns holds(default), then applies the same default to a penguin
and returns withdrawn(exception(penguin_exception)) — the default is retracted
because the flightless exception fires. The exception is named in the
justification. Pass criterion met: "birds fly" withdrawn for penguin with
exception named.


INTRODUCTION

Classical logic is monotonic: adding new facts never invalidates previous
conclusions. But common sense reasoning is non-monotonic. We assume birds fly
until we learn that a particular bird is a penguin, at which point we retract
the conclusion. Non-monotonic (defeasible) reasoning formalises this retraction
pattern.

Rung 17 provides a dedicated defeasible reasoning module with full exception
handling, going beyond the preview at Rung 1 to expose the full retraction
mechanism with named exceptions and rule chain.


BACKGROUND

Non-monotonic reasoning was introduced by McCarthy (1980) as circumscription and
by McDermott and Doyle (1980) as non-monotonic logic. Reiter's Default Logic
(1980) provides a formal framework for default rules with exceptions.

Mentova's implementation follows the closed-world default pattern:
  1. Apply the default if the triggering condition holds.
  2. Check all exception rules; if any exception condition holds, retract.
  3. Report either holds(default) or withdrawn(exception(Name)).


METHOD

A new module, src/mentova/nonmonotonic.pl, implements:

    mentova_defeasible(+Query, -Result, -Justification)

The core predicate defeasible_conclusion/3 checks the default_rule/2 condition
and then checks all exception_rule/3 entries. If an exception fires, its name
is recorded in the withdrawn/1 status and in the justification.

Query forms:

    query(Prop)             defeasible status of Prop
    exceptions(Prop)        list all exceptions for Prop
    is_exception(Prop, X)   is X an exception case?

Top-level clause:

    % Query Mentova for nonmonotonic reasoning about 'NMQuery'.
    mentova_query(nonmonotonic, NMQuery, answer(Result, Just)) :-
        % State the fact: mentova defeasible(...).
        mentova_defeasible(NMQuery, Result, Just).


DEMONSTRATION

Query 1 — flies(canary):

    % Query Mentova for nonmonotonic reasoning about 'query(flies(canary))'.
    ?- mentova_query(nonmonotonic, query(flies(canary)), R).
    R = answer(holds(default),
               just(defeasible(flies(canary), default(bird_flies),
                               no_exception_found, conclusion(flies))))

    Default "birds fly" applies to canary. No exception fires. Status: holds.

Query 2 — flies(penguin):

    % Query Mentova for nonmonotonic reasoning about 'query(flies(penguin))'.
    ?- mentova_query(nonmonotonic, query(flies(penguin)), R).
    R = answer(withdrawn(exception(penguin_exception)),
               just(defeasible(flies(penguin), default(bird_flies),
                               exception(penguin_exception), conclusion(does_not_fly))))

    Default applies (penguin is_a bird). But exception fires:
    has_property(penguin, flightless) triggers penguin_exception.
    Default is retracted. Status: withdrawn. Exception name: penguin_exception.


RESULTS

The default "birds fly" is correctly applied to canary and correctly withdrawn
for penguin, with the exception name (penguin_exception) named in the justification.

Pass criterion met: "birds fly" withdrawn for penguin with exception named.


SIGNIFICANCE

Non-monotonic reasoning is one of the most practically important reasoning modes.
Real-world knowledge is full of defaults that admit exceptions: employees get paid
monthly (except contractors), customers pay before delivery (except trusted partners),
birds fly (except penguins, emus, and kiwis).

Mentova's defeasible engine handles these patterns cleanly, reporting not just the
conclusion but which rule was retracted, which exception fired, and what name the
exception carries.


HONEST LIMITATIONS

The implementation uses Prolog's built-in negation-as-failure (\+) to check for
the absence of exceptions. This is a closed-world assumption: if an exception is
not found, it is assumed not to exist.

Multiple exceptions are handled by collecting them all and using the first one found.
A priority ordering among exceptions is not implemented.

Interaction between multiple defaults (e.g., default-A overrides default-B overrides
default-C) would require an explicit conflict-resolution mechanism (e.g., specificity
or recency ordering).

Requirements: SWI-Prolog 9.0.4 or later.

  git clone https://github.com/ai-university-aiu/Mentova.git
  cd Mentova
  % Load the Mentova bootstrap into SWI-Prolog, execute the query, and halt.
  swipl -l src/mentova/mentova.pl \
        % Supply a goal or termination instruction to the SWI-Prolog command.
        -g "mentova_boot, \
            % Query Mentova for nonmonotonic reasoning about 'query(flies(canary))'.
            mentova_query(nonmonotonic, query(flies(canary)), R1), write(R1), nl, \
            % Query Mentova for nonmonotonic reasoning about 'query(flies(penguin))'.
            mentova_query(nonmonotonic, query(flies(penguin)), R2), write(R2), nl" \
        % Supply a goal or termination instruction to the SWI-Prolog command.
        -t halt
