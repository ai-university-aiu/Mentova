Abstract

PrologAI gains an objcomp pack (14 oc_* predicates, Layer 167) for adjacency
and connected-component analysis over collections of obj(Color, Cells) terms.
Two objects "touch" when any cell of the first is 4-adjacent (Manhattan distance
1) to any cell of the second. The pack provides: the touching test, pair
enumeration, adjacency lists, touch degree, isolation detection, BFS-connected
component extraction, and aggregate queries over components and degrees.

40 of 40 PLUnit tests pass.

This work package is PrologAI WP 188, merged as PR 273 on 2026-06-26.

1. Motivation

Reasoning about how objects relate to each other spatially is a core requirement
for visual problem solving. "Which objects are adjacent?" and "which objects form
a connected cluster?" appear constantly. Before this pack:

- Testing if two specific objects touched required writing cell-pair iteration
- Finding which objects touch a given object was available in the proximity pack
- But finding all connected groups of mutually-reachable objects required custom BFS

The objcomp pack provides:

    oc_components(Objs, Components),  % all touching-connected groups
    oc_isolated(Objs, Isolated),       % objects that touch nothing
    oc_largest_component(Objs, Cluster). % the largest touching cluster

2. Core Concepts

2.1 The Touching Relation

oc_touches(O1, O2) succeeds when any cell of O1 is 4-adjacent to any cell of O2.
4-adjacency means: same row and adjacent columns (|C1-C2| = 1), or same column
and adjacent rows (|R1-R2| = 1). Diagonal adjacency is not included.

The predicate cuts after finding the first adjacent cell pair, making it
efficient for large objects with many cells.

2.2 Connected Components

Two objects are in the same connected component if there exists a chain of
pairwise-touching objects between them. This is the standard connected-component
concept, now applied at the object level rather than the cell level.

oc_connected(Obj, Objs, Component) uses BFS:
1. Start with [Obj] in the queue and [] in visited
2. Dequeue the head; if already visited, skip
3. Otherwise add to visited and enqueue all unvisited touching neighbors
4. Return the accumulated visited list

oc_components extracts all components by recursively:
1. Take the first remaining object, find its component
2. Remove all component members from the remaining list
3. Repeat until no objects remain

3. Predicate Descriptions

3.1 Touching

oc_touches(+Obj1, +Obj2): the 4-adjacency test. For each pair of cells (one from
each object), check if they are orthogonally adjacent. Cut after first success.

3.2 Pairs and Adjacency

oc_touching_pairs(+Objs, -Pairs): pairs where O1's index < O2's index in Objs,
determined via nth0/3. This avoids the == operator ambiguity with unbound variables
that would arise from a recursive structural-match helper.

oc_adj_list(+Objs, -AdjList): Obj-[Neighbors] for each Obj. Built via findall
over all other objects that pass oc_touches.

oc_degree(+Obj, +Objs, -N): count of touching neighbors via findall + length.

3.3 Isolation

oc_isolated(+Objs, -Isolated): findall objects where \+ exists touching neighbor.
The \+ forall pattern: succeed iff no neighbor exists.

3.4 Connected Components

oc_connected(+Obj, +Objs, -Component): BFS with two accumulators: Queue (objects
to explore) and Visited (objects already explored). Base: Queue = [] -> return
Visited. Recursive: if head is already visited, skip; otherwise add to Visited
and enqueue all unvisited touching neighbors.

oc_components(+Objs, -Components): partition through repeated oc_connected calls.
Uses subtract/3 to remove component members from the remaining list.

oc_num_components(+Objs, -N): length of oc_components result.

3.5 Component Selection

oc_largest_component: msort on N-C pairs (N = component size) then last/2.
oc_smallest_component: msort then first element.
oc_singleton_components: filter components with length 1.
oc_shared_components: filter components with length > 1.

3.6 Degree Aggregates

oc_max_degree: findall all degrees then max_list.
oc_sort_by_degree: keysort on D-Obj pairs; stable (preserves input order for ties).

4. Design Decisions

4.1 nth0 for Pair Enumeration

The initial implementation used a recursive helper with H == A to find ordered
pairs. This failed because == tests structural identity: an unbound variable O1
is not identical to a ground term, so `H == O1` was always false during the
findall. Using nth0/3 indices to enforce I < J works correctly with unbound
variables by position comparison rather than structural identity.

4.2 BFS over DFS for Connected Components

BFS ensures that all objects at "distance 1" from the start are found before
objects at "distance 2," giving a predictable exploration order. DFS would give
the same final result but a different intermediate ordering. BFS is slightly
more natural when the goal is "find all reachable objects."

4.3 No Dependency on Proximity Pack

Although the proximity pack has px_touching (which also tests 4-adjacency), the
objcomp pack implements oc_touches independently. This avoids a cross-pack
dependency and keeps Layer 167 self-contained.

5. Examples

Example 1: finding all connected groups

    A = obj(a,[r(0,0)]), B = obj(b,[r(0,1)]), C = obj(c,[r(5,5)]),
    oc_components([A,B,C], Comps).
    % Comps = [[A,B], [C]]  (A and B touch; C is isolated)

Example 2: finding isolated objects

    oc_isolated([A,B,C], Isolated).
    % Isolated = [C]

Example 3: degree of each object

    oc_degree(A, [A,B,C], D).  % D = 1 (A touches B only)
    oc_degree(B, [A,B,C], D).  % D = 1 (B touches A only)
    oc_degree(C, [A,B,C], D).  % D = 0 (C is isolated)

Example 4: the largest cluster

    oc_largest_component([A,B,C,D,E,F], Cluster).
    % Returns the component containing A,B,D,F,C (5 objects)

The objcomp pack gives PrologAI a clean interface for reasoning about which
objects cluster together through spatial adjacency. Connected-component analysis
at the object level - finding groups, isolated objects, hub objects, and cluster
sizes - is now available in a single predicate call.
